# 2026-03-12 对话记录：把 DeepSeek 接成 OpenClaw 可调度模型

## 一、本次对话我与你的聊天记录总结

这轮对话的目标非常明确：你要求我把 `DeepSeek` 真正接成这版 OpenClaw 可调度的模型，而不是继续停留在“auth profile 已存在、但实际调度层里没有它”的半成状态。

这件事和前一轮 `Moonshot + Qwen` 的差别在于：

1. `Moonshot` 和 `Qwen` 在这版 OpenClaw 里都有比较清晰的 provider 注入路径；
2. `DeepSeek` 虽然有 `deepseek:manual` 这个 auth profile，但自动 provider 注入器并不会像对 `moonshot` 那样，自动把它挂进 runtime model catalog；
3. 所以这轮真正要做的不是“再贴一次 key”，而是找出这版 OpenClaw 真正支持的接法，然后用那条路径把 `DeepSeek` 接到调度层。

### 本轮定位过程

我先做了 3 层核对。

#### 1. 认证层是否存在

通过：

```bash
openclaw config get auth.profiles
```

我确认到：

- `deepseek:manual`

确实已经存在。

这说明你前面自己接的 DeepSeek API 并不是失败了，而是停在了认证层。

#### 2. 自动 provider 注入器是否支持 `deepseek`

我继续检查了这版 OpenClaw 的内置 provider 注入逻辑。结果发现：

- 它会自动注入 `moonshot`
- 会自动注入 `qwen-portal`
- 也会自动注入 `together`、`openrouter`、`huggingface`、`qianfan` 等
- 但没有自动注入 `deepseek`

这就是为什么：

```bash
openclaw models list --all --provider deepseek --plain
```

之前会返回：

```text
No models found.
```

问题不在 token，而在于：  
**这版 OpenClaw 不会仅凭 `deepseek:manual` auth profile，就自动给你生成一个 `deepseek` provider runtime catalog。**

#### 3. 这版是否支持“手动 provider 配置”路径

答案是支持。

我继续核对了本地文档和配置参考，确认这版 OpenClaw 支持通过：

- `models.mode`
- `models.providers.<providerId>`

手动定义 OpenAI-compatible provider。

这正好适合 DeepSeek，因为 DeepSeek 官方 API 提供 OpenAI-compatible 路径。

### 本轮采用的接法

我最终采用的是：

- 手动新增 `models.providers.deepseek`
- 使用 DeepSeek 官方 OpenAI-compatible endpoint
- 在 provider 下显式注册两个模型：
  - `deepseek-chat`
  - `deepseek-reasoner`

这样做的意义是：

1. 不依赖这版 OpenClaw 缺失的自动注入逻辑；
2. 直接把 DeepSeek 变成一个运行时可见、可列出、可进入 fallback 链的正式 provider；
3. 后续你即使继续扩展多 agent，也不会再受“auth 有了但 catalog 没出来”的限制。

### 本轮实际落地结果

它们的别名分别是：

- `deepseek`

这一步的意义是：  
它们不再只是 provider 内部定义，而是进入了 agent 调度允许集合。

#### 3. DeepSeek Chat 已进入正式 fallback 链

当前默认链路已经变成：

也就是说，`deepseek-chat` 现在已经是一个真正可被 fallback 选中的可调度模型。

#### 4. 多 agent 的 fallback 也已经补入 DeepSeek Chat

我没有把 `DeepSeek` 只塞进全局，而是让多 agent 结构也认它：

- `learning`

  - fallback 里加入 `deepseek/deepseek-chat`
- `curator`
  - `moonshot` 主

- `executor`

- `validator`
  - `qwen` 主
  - fallback 里加入 `deepseek/deepseek-chat`

这意味着它已经不仅是“能被看见”，而是已经真正参与到多 agent 协作的后备调度中。

#### 5. DeepSeek Reasoner 也已注册，但没有被我粗暴塞进默认主链

这是有意为之。

我让它进入了：

- provider catalog
- agent 默认模型白名单

但没有让它成为工具型 agent 的默认主模型，也没有把它直接写进全局 fallback 主链。

原因是：

1. `reasoner` 更适合纯推理、长链思考、复盘、战略类场景；
2. 这轮的目标是先让 `DeepSeek` 稳定“接进来”；
3. 工具链兼容、函数调用、复杂 agent 执行这类问题，我不应该在没有独立 smoke test 的情况下直接把 `reasoner` 压到默认链上。

所以这轮的策略是：

- `deepseek-chat`：正式进入调度链
- `deepseek-reasoner`：正式注册，但先不强塞到默认主链

这是更稳健的做法。

### A. 运行态配置修改（不进 Git）
