# 2026-03-13 对话记录

## 一、本次对话我与你的聊天记录总结

这轮你的要求很明确：不要停在“管理模式已经有状态机”这一步，而是继续把它推进到真正可执行的多 agent 管理层。你同意我上一轮的判断，也就是接下来最有价值的方向，不是再加更多概念，而是把 `ready` branch 真正接到 owner 的执行入口，让 `learning / curator / executor / validator` 不再只是 Task Tree 里的名字，而是能被分发、留痕、检查的工作对象。

我这轮首先重新核对了当前 OpenClaw 现状，确认了几件关键事实：

1. 现有第 8 / 9 / 10 步已经有 Task Tree、dispatch、blackboard、validator、approval 这些管理骨架。
2. `strategy_review` route 已经是正式 route，且默认绑定 `deepseek/deepseek-reasoner + low_tool`。
3. 当前最大缺口不是“没有规则”，而是“没有 branch handoff 执行层”。
4. 这版 OpenClaw 本身确实存在 `openclaw agent`、`openclaw agents`、`openclaw system event`、`openclaw cron` 等入口，但现有管理链还没有把 `ready_branches` 自动翻译成统一的 branch packet 与透明活动日志。

基于这个判断，我没有去硬写一个假装“已经自动跑起来”的黑箱多 agent 调度器，而是落了一个更稳的执行层：

- `dispatch` 之后，ready branch 会先生成正式 handoff packet
- packet 会统一写到 `shared/runtime/dispatch/<task_id>/<branch_id>.json`
- handoff 事件会统一写到 `shared/runtime/activity/<task_id>.jsonl`
- 黑板状态会从 `ready` 推进到 `assigned`
- packet 里还会包含推荐调用方式，也就是后续如果要真的让子 agent 自动运行，可以直接复用 `openclaw agent --agent <owner>` 这一层

这一步的核心意义是：

1. 让你的“软隔离多 agent 管理模式”第一次有了真正的 branch 工作指令
2. 让所有 agent 的后台工作具备统一格式留痕，而不是散落在不同地方
3. 让 `strategy_review` 不只是注册成模型或 route，而是进入真实 handoff 体系

### 2. `shared/runtime/management/task_dispatch_lib.mjs`

这是本轮最核心的代码修改。

我在这里补了完整的 handoff 执行层，具体包括：

#### a. 扩展 dispatch state

`computeDispatchState()` 现在不只是返回：

- `branch_id`
- `owner`
- `status`
- `route`
- `tool_mode`
- `model_hint`

还会携带：

- `goal`
- `expected_output`
- `acceptance`

作用是：

1. 后续 handoff packet 能直接拿到完整 branch 上下文
2. 不需要在 packet 生成阶段再去拼接缺失信息

#### b. 队列路径支持配置化

新增：

- `resolveQueueDir()`
- `writeTaskTreeSnapshot(root, taskTree, config)`

作用是：

1. queue 不再写死到单一目录
2. smoke test 时可以把 queue 重定向到 `/tmp`
3. 以后如果你想把管理运行态拆出去，也有路径层面的余地

#### c. 新增 dispatch / activity 目录解析

新增：

- `resolveDispatchDirs()`

作用是：

1. 统一解析 `dispatchDir` 和 `activityDir`
2. 后续 workflow、CLI、测试都可以共用同一套路径逻辑

#### d. 新增 branch packet 构建能力

我新增了：

- `buildBranchPacket()`
- `buildBranchInstructions()`
- `branchSessionKey()`
- `ownerWorkspacePath()`

生成的 packet 会包含：

- `task_id`
- `title`
- `branch` 元信息
- `inputs.task_tree_path`
- `inputs.blackboard_card_path`
- `execution.workspace_path`
- `execution.session_key`
- `execution.preferred_model`
- `execution.recommended_invocation`
- `output_contract`
- `transparency.activity_log_path`
- `instructions`

作用是：

1. 真正把 ready branch 变成“可执行工作单”
2. 让后续子 agent 拿到的是明确、边界清楚的 packet，而不是一句口头交办
3. 把模型偏好、低工具约束、黑板更新责任一起显式写进去

#### e. 新增统一活动日志

我新增了：

- `appendActivityLog()`

所有 handoff 事件会统一以 JSONL 方式写入：

- `shared/runtime/activity/<task_id>.jsonl`

日志字段至少包含：

- `timestamp`
- `task_id`
- `branch_id`
- `owner`
- `route`
- `tool_mode`
- `model_hint`
- `event`
- `status`
- `packet_path`

作用是：

1. 满足你最早提出的“所有 agent 后台活动状态保持公开透明”
2. 让后续 OpenClaw 自己也能统一检索这些活动历史
3. 给未来设备接入、主动监督、任务回溯提供结构化运行痕迹

#### f. 新增 handoff 主函数

我新增了：

- `handoffReadyBranches()`

它会做以下几件事：

1. 读取并校验 Task Tree
2. 找出 `ready_branches`
3. 为每个 ready branch 生成 packet
4. 追加 activity log
5. 如果黑板存在，则把状态从 `ready` 推到 `assigned`
6. 输出 handoff 摘要 JSON

作用是：

1. 把第 8 / 9 / 10 步从“能标出 ready 分支”推进到“能正式分发”
2. 让 branch handoff 变成统一、可重复调用的 runtime 能力
