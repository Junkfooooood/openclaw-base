# 如何正确登陆和切换微信

## 一、先搞清楚这件事在做什么

当前这套微信接入，不是把你的主号“授权给机器人临时用一下”，而是：

- 让某一个微信号，正式成为 OpenClaw 在微信里的消息入口。

也就是说：

- 谁扫二维码登录，谁就是“机器人号”
- 别人以后给这个号发消息，OpenClaw 才能收到
- OpenClaw 回消息，也会从这个号发出去

因此建议：

- 优先使用副号 / 测试号
- 不要一开始就直接用主号接

## 二、如何正确登录微信号

### 第 1 步：先取当前二维码链接

在 Mac mini 终端执行：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/latest_wechat_qr.sh
```

它会输出一个二维码页面链接。

### 第 2 步：打开二维码页面

把终端输出的链接复制到浏览器打开。

如果旧二维码失效，就重新执行上面的命令，取最新的链接。

### 第 3 步：用“准备作为机器人号的微信”去扫码

这一点最关键：

- 扫码的那个微信号，就是以后机器人在微信里的身份。

不是“你本人主号授权一下”，而是“这个号本身成为机器人号”。

### 第 4 步：在手机上确认登录

扫码后，如果微信弹出确认或授权提示，需要在手机上明确点确认。

只扫码不确认，通常不会真正完成登录。

### 第 5 步：等待同步

确认后，等待约 `5-15` 秒，让：

- WeChatPadPro
- OpenClaw gateway
- wechat channel monitor

把登录态同步完成。

### 第 6 步：用另一个微信号发起测试

不要让机器人号自己和自己说话。

应该用“另一个微信号”去给机器人号发消息。

第一条建议发：

```text
你好
```

当前这台机器的 live 验收配置里，为了让“第一次私聊就能收到配对码”，已经临时移除了首条私聊的前缀限制。

因此首轮验收时：

- 直接发纯文本 `你好` 最稳
- 等配对完成后，再恢复习惯性的 `@ai ...` 也可以

### 第 7 步：如出现配对码，就批准

当前私聊策略是：

- `dmPolicy = pairing`

所以第一次有人私聊机器人号时，系统可能会回一个配对码。

此时在 Mac mini 终端执行：

```bash
openclaw pairing approve wechat <配对码>
```

批准后，再发一次：

```text
@ai 你现在是谁
```

若正常回复，则说明微信接入成功。

## 三、如何判断是否真的登录成功

可在终端执行：

```bash
grep '\[wechat\]' ~/.openclaw/logs/gateway.log | tail -n 30
```

若看到类似内容，通常说明登录已成功或接近成功：

- `login confirmed`
- `starting monitor`
- `polling for messages`

若一直反复出现：

- `waiting for QR code scan`
- `login not confirmed — will retry`

那通常表示：

- 还没扫码
- 扫了但没确认
- 或二维码已过期

## 四、以后如何切换微信号

“换号”本质上就是：

- 让 WeChatPadPro 不再使用当前已登录微信
- 然后重新显示一个新的登录二维码
- 再让另一个微信号去扫

### 最简切换法

如果只是想从当前号切到另一个号，先执行：

```bash
docker restart openclaw-wechatpadpro
```

然后重新取二维码：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/latest_wechat_qr.sh
```

再用新的微信号扫码。

### 更稳的切换法

如果你怀疑旧登录态残留，建议直接重新拉起整套栈：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/start_wechat_stack.sh
```

然后再执行：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/latest_wechat_qr.sh
```

### 切换后的验收方式

切换新号后，仍然按登录验收流程走：

1. 用另一个号给机器人号发：

```text
@ai 你好
```

2. 若要求配对，则批准：

```bash
openclaw pairing approve wechat <配对码>
```

3. 再发：

```text
@ai 你是谁
```

## 五、最常见的误区

### 误区 1：我扫的是“授权给机器人使用我的主号”

不是。

更准确地说：

- 你扫上的那个号，就是机器人号本身。

### 误区 2：登录后要从同一个号给自己发消息测试

不对。

应该使用另一个微信号去找机器人号聊天。

### 误区 3：发普通消息就会回复

不一定。

当前这台机器当前的 live 配置中，首条私聊验收已经不强制 `@ai`。

但如果以后又恢复了 `triggerPrefix`，那么不带前缀的消息仍可能被忽略。

### 误区 4：二维码过期了还能继续扫

不行。

二维码过期后，应重新执行：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/latest_wechat_qr.sh
```

### 误区 5：状态显示在线，就说明一定还能正常收消息

不一定。

这套微信桥偶尔会出现一种“半在线”状态：

- `GetLoginStatus` 仍返回在线
- 但 `gateway.log` 长时间不再出现新时间戳
- 机器人也收不到新的文本私聊

这种情况下，不要只盯着手机里是否显示 `iPad 登录`，而应按下面的方法判断：

1. 看当前日志是否仍在刷新：

```bash
grep '\[wechat\]' ~/.openclaw/logs/gateway.log | tail -n 30
```

2. 若日志时间长时间停住，可先重启 `gateway`：

```bash
openclaw gateway restart
```

3. 若重启后仍然“显示在线但不收消息”，优先把它当作“半在线假活跃”，直接重拉微信栈并重新扫码最稳：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/start_wechat_stack.sh
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/latest_wechat_qr.sh
```

## 六、如何自动重连

现在仓库里已经补了一套本地看门脚本与后台守护：

- `shared/runtime/wechat/wechat_watchdog.mjs`
- `shared/runtime/wechat/install_wechat_watchdog.sh`

它的工作方式是：

1. 定期查询微信桥 `GetLoginStatus`
2. 若明确掉线或接口不可达：
   - 重启 `gateway`
   - 重启微信桥
   - 再重启一次 `gateway`
3. 若仍需要人工登录，则自动把最新二维码写入：
   - `shared/runtime/wechat/state/latest-recovery-qr.txt`
   - `shared/runtime/wechat/state/watchdog-status.json`

### 后台自动守护

安装后，系统会每 `5` 分钟跑一次：

```bash
bash /Users/linqingxuan/.openclaw/shared/runtime/wechat/install_wechat_watchdog.sh
```

### 手动一键自愈

如果你觉得当前是“半在线假活跃”，也可以手动强制跑一轮：

```bash
node /Users/linqingxuan/.openclaw/shared/runtime/wechat/wechat_watchdog.mjs heal --force
```

### 自动重连的边界

要如实说清：

- 程序可以自动完成“检测、重启、拉起二维码”
- 但如果微信桥要求重新登录，最后那一下仍需要人工扫码

也就是说，这套自动重连现在能做到的是：

- 自动自愈 clear disconnect
- 自动产出新的恢复二维码
- 把人工步骤压缩到“只剩扫码”

取最新链接。

## 六、当前推荐做法

如果你现在只是测试，推荐这样做：

1. 先用副号扫登录
2. 用另一个号发 `@ai 你好`
3. 若出现配对码，就批准
4. 验收通过后，再决定是否长期保留这个号作为机器人号

一句话总结：

- 扫码的号 = 机器人号
- 测试消息要从另一个号发
- 第一条先发 `@ai 你好`
- 要换号就重启 `wechatpadpro` 后重新扫码
