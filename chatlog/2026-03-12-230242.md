1. 把 `Moonshot` 和 `Qwen` 从“认证层已存在”推进到“agent 可调度模型层已存在”；
2. 给多 agent 协作设定一个合理的模型分工基线；

### 本轮定位过程

我先重新核对了当前 live 配置和本机模型 catalog，得到以下关键事实：

#### 1. 认证层已经有了

通过：

```bash
openclaw config get auth.profiles
```

我确认到以下 profile 已经存在：

- `deepseek:manual`
- `moonshot:manual`
- `qwen-portal:manual`

这说明你前面自己接的 API 并不是白做，认证层确实已经存在。

#### 2. 但调度层还没有

通过：

```bash
openclaw config get agents.defaults.models
```

我确认到当时默认模型白名单里仍然只有：

- `openai-codex/gpt-5.4`
- `openai/gpt-5.4`
- `openai/gpt-5.3-codex`

这就解释了为什么 `openclaw models status --plain` 仍然只显示 OpenAI 主模型：  
因为系统还没把 `Moonshot/Qwen` 纳入 agent 默认可调用模型集合。

#### 3. 本机 catalog 对三家 provider 的支持程度不一样

我继续核对了本机 catalog：

```bash
openclaw models list --all --provider moonshot --plain
openclaw models list --all --provider qwen-portal --plain
openclaw models list --all --provider deepseek --plain
```

得到的结果是：

- `moonshot`：能识别 `moonshot/kimi-k2.5`
- `qwen-portal`：能识别 `qwen-portal/coder-model`、`qwen-portal/vision-model`
- `deepseek`：返回 `No models found.`

这意味着：

- `Moonshot` 和 `Qwen` 已经具备直接纳入调度层的条件；
- `DeepSeek` 虽然 auth profile 已经存在，但在这版本地 OpenClaw 里，没有给出一个可以直接落进 `agents.defaults.models` 的稳定 `deepseek/<model>` 目录；
- 所以如果我现在硬把 `DeepSeek` 写进默认调度链，风险是“配置上去了，但实际上不可解析或不可调度”。

### 本轮实际落地结果

我已经在 live 配置层完成了 `Moonshot + Qwen` 的接入，并重启了 gateway 让配置生效。

生效后的核心结果是：

#### 1. 默认主备链已经变成

- 主模型：`openai-codex/gpt-5.4`
- fallback 顺序：
  - `moonshot/kimi-k2.5`
  - `qwen-portal/coder-model`
  - `openai/gpt-5.4`
  - `openai/gpt-5.3-codex`

这个顺序的作用是：

1. 继续保留你当前最稳的 OpenAI Codex 做主链；
2. 当 OpenAI 侧限流、异常、额度紧张时，优先切到跨厂商的 `Kimi` 与 `Qwen`；
3. 把 OpenAI 的另外两个模型保留在更后面，作为同厂商内部兜底。

#### 2. 多 agent 的模型职责基线已经定下

我给各 agent 设了如下运行基线：

- `main`
  - 继续继承默认链，即以 `openai-codex/gpt-5.4` 为主
- `learning`
  - `moonshot/kimi-k2.5` 为主
  - 回退到 `qwen-portal/coder-model`、`openai/gpt-5.4`
- `curator`
  - `moonshot/kimi-k2.5` 为主
  - 回退到 `qwen-portal/coder-model`、`openai/gpt-5.4`
- `executor`
  - `openai-codex/gpt-5.4` 为主
  - 回退到 `qwen-portal/coder-model`、`moonshot/kimi-k2.5`、`openai/gpt-5.3-codex`
- `validator`
  - `qwen-portal/coder-model` 为主
  - 回退到 `moonshot/kimi-k2.5`、`openai/gpt-5.4`

这个分工不是硬约束，而是运行基线。它的意义是：

1. 让 `learning/curator` 这种偏长文本整理、知识归纳、中文学习陪伴类任务，优先走 `Kimi`；
2. 让 `validator` 这种偏结构化检查、规则复核、格式审阅的任务，优先走 `Qwen Coder`；
3. 让 `executor` 继续保持在 Codex 主链上，保证代码和自动化执行侧的稳定性；
4. 保持 `main` 继续做总控，不因为接入新模型就失去你当前最熟悉的主入口行为。

这说明 `Moonshot` 和 `Qwen` 已经不只是 auth profile 存在，而是已经进入当前 agent 运行时的可调度模型集合。

### 关于 DeepSeek，这轮为什么我没有硬接进去

这一步我有意保守处理。

原因不是我忘了，而是我检查之后，发现这版 OpenClaw 的本地 provider 支持状态不够清晰：

1. `auth.profiles` 里确实有 `deepseek:manual`
2. 但 `openclaw models list --all --provider deepseek --plain` 返回 `No models found`
3. 全局大 catalog 里虽然能看到很多 `DeepSeek` 家族模型，但它们挂在的是：
   - `openrouter/...`
   - `huggingface/...`
   - `vercel-ai-gateway/...`
   - `amazon-bedrock/...`
   这些 provider 下

也就是说，目前我能负责任确认的是：

- 你“有 DeepSeek 认证入口”
- 但我还不能确认你“有可稳定调度的裸 `deepseek/<model>` 入口”

所以这轮我选择：

- 先把 `Moonshot + Qwen` 彻底接好；
- 把 `DeepSeek` 保持在“已认证、未正式并入默认调度链”的状态；
- 等后面单独做一次更明确的 provider 验证，再决定它应该走哪种接法。

这是为了避免把一个表面上“好像可用”、实际上运行时不稳定的模型写进管理主链，反而破坏你现在已经逐渐稳定下来的多 agent 管理模式。

### C. 本轮刻意没有提交的文件

这轮我刻意保留了两个运行态脏文件，不让它们混进 Git：
