# 2026-03-12 对话记录：模型认证层与模型白名单层核对

## 一、本次对话我与你的聊天记录总结

本轮对话的核心问题是：你在与我继续配置多模型之前，就已经手动接入了 `deepseek`、`moonshot`、`qwen` 三家的 API，但执行 `openclaw models status --plain` 时，输出仍然只有：

```text
openai-codex/gpt-5.4
```

你希望确认这到底代表什么，是否说明前面的接入没有成功。

这次我没有直接改你的模型调度配置，而是先做了状态核对，目的是把问题定位清楚，避免在认知不完整的情况下继续写配置，导致后面更乱。

本次核对得到的结论是：

1. 你接入的三家 provider 的**认证层**已经存在，并不是完全没接上。
2. 但 OpenClaw 当前真正用于 agent 调度的**模型白名单层**里，仍然只有 OpenAI 的几个模型。
3. 因此现在系统的真实状态是：  
   `provider auth 已存在`，但 `agent 可调用模型列表` 还没有同步补进去。
4. 另外，你执行的 `openclaw models status --plain` 在这版本地行为上，更像是“回显当前主模型”，而不是“列出所有已认证 provider 和模型”。

我实际核对到的本地状态如下：

### 1. 认证层 `auth.profiles`

命令：

```bash
openclaw config get auth.profiles
```

结果中已经包含：

- `deepseek:manual`
- `moonshot:manual`
- `qwen-portal:manual`

这说明三家的 token/profile 确实已经登记进了 OpenClaw 的认证配置层。

### 2. 模型白名单层 `agents.defaults.models`

命令：

```bash
openclaw config get agents.defaults.models
```

结果里仍然只有：

- `openai-codex/gpt-5.4`
- `openai/gpt-5.4`
- `openai/gpt-5.3-codex`

也就是说，虽然 provider 已经接进来，但 agent 默认允许调用的模型仍然只有 OpenAI。

### 3. 本机当前可识别到的 provider 模型目录

我继续核对了本机 catalog：

```bash
openclaw models list --all --provider moonshot --plain
openclaw models list --all --provider qwen-portal --plain
openclaw models list --all --provider deepseek --plain
```

结果是：

- `moonshot`：可识别 `moonshot/kimi-k2.5`
- `qwen-portal`：可识别 `qwen-portal/coder-model`、`qwen-portal/vision-model`
- `deepseek`：当前返回 `No models found.`

这说明：

- `Moonshot` 和 `Qwen` 已经具备继续写入调度配置的基础。
- `DeepSeek` 目前在这版本地 OpenClaw 里，至少从本机 catalog 视角看，还没有直接暴露出可选模型目录，所以后面不能简单照抄前两家处理。

### 4. 为什么 `openclaw models status --plain` 只显示一个模型

原因不是“你前面没有接入成功”，而是以下两点叠加：

1. `status --plain` 这版输出的重点是**当前主模型**；
2. 当前主模型仍然是 `openai-codex/gpt-5.4`，而你还没有把 `moonshot/qwen` 正式加入 agent 的默认模型集合，更没有设为 fallback 或主模型。

所以这条输出目前只能说明：

- 当前主模型还是 OpenAI；
- 不能用它单独判断 `moonshot/qwen/deepseek` 是否已经完成“全链路接入”。

更准确的判断方法是同时看两层：

1. `openclaw config get auth.profiles`
2. `openclaw config get agents.defaults.models`

只有两层都补齐了，才算是真正能参与多模型协作。
