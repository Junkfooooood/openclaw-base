# 2026-03-12 对话记录

## 一、本次对话总结

本轮在不增加硬约束的前提下，正式把“轻量 route review loop”落到了系统里。

核心目标是：

- 不把 route 分发变成死规则；
- 但让 validator 可以在 route 产出后做一个非常轻的复核；
- 把复核结果以 advisory 的方式反馈给系统，作为后续自迭代的素材。

本轮完成的实际工作包括：

1. 新增一份轻量 SOP：
   - `shared/sop/active/Route_Review_Loop_SOP_v1.md`
   - 用来定义 route review 的目的、输入格式、输出标准、复核原则和自迭代关系。

2. 扩展 `policy-gate` / `validator-run`
   - 在不破坏旧工作流校验的前提下，新增了 `route-review` 模式。
   - 现在 `validator-run` 默认仍然是原来的 workflow 结构校验；
   - 只有显式传 `--kind route-review` 时，才走 route 复核逻辑。

3. 增加了 route review 的 workflow 示例
   - `shared/workflows/route_review.prose.md`
   - 这个文件不是硬流程，而是一个可复用样板，告诉未来的 agent：route 方案产出后，如何交给 validator 做轻量复核。

4. 补了能力可见性
   - 在 `MEMORY.md`、`workspace-validator/AGENTS.md`、`Validation_Rules.md` 中补上了 route review 的存在和边界；
   - 这样 agent 不需要“从代码猜功能”，而是能在上下文里直接知道这件事已经存在。

5. 做了双重验证
   - 旧的 `validator-run` 仍然正常 PASS，没有被改坏；
   - 新的 `validator-run --kind route-review` 会返回 `PASS + advisory`，例如“可考虑补 logs”，而不会把它直接判成失败。

## 二、修改了什么文件，作用是什么

### `shared/sop/active/Route_Review_Loop_SOP_v1.md`

这是本轮新增的核心协议文件。

它的作用是：

1. 定义轻量 route review 的目标
   - 明确这不是审批闸门；
   - 不是阻断器；
   - 而是一个“route 是否大致合理”的低成本检查。

2. 明确 route review 的输入格式
   - 主要输入包括：
     - `source_summary`
     - `route_plan`
   - 这样后续任何 agent 想调用 route review，都知道应该给 validator 什么材料。

3. 明确 route review 的输出口径
   - 仍然只允许 `PASS / FAIL / BLOCK`
   - 但特意强调：
     - “可探索空间”优先给 advisory
     - 不要轻易 `FAIL`

4. 把 `logs` 的地位定义成“默认基线但非硬约束”
   - 这是非常关键的一点；
   - 你明确不想把 route 规则写死，所以我这里把“没有 logs”定义成 advisory，而不是 fail 条件。

5. 把 route review 和自迭代串起来
   - 这份 SOP 里明确写了：
     - `对话 -> route 方案 -> route review -> advisory 累积 -> SOP 更新`
   - 这就把“如何软迭代”写成了系统可复用的方法，而不是只停留在聊天里。

### `extensions/policy-gate/index.mjs`

这是本轮最主要的实现改动。

它的作用分成 4 块：

1. 新增 route 复核逻辑
   - 增加了 route 名标准化和 route plan 校验函数；
   - 现在系统知道哪些 route 是合法的：
     - `logs`
     - `knowledge`
     - `taskboard`
     - `capabilities`
     - `reputation`
     - `strategy`

2. 新增 route alias 归一化
   - 比如：
     - `日志` -> `logs`
     - `知识库` -> `knowledge`
     - `任务看板` -> `taskboard`
   - 这样 route review 不会因为中英文混用而误判。

3. 给 `validator-run` 增加 `--kind route-review`
   - 默认不传时，仍然走原有 workflow payload 校验；
   - 只有显式传 `route-review` 才走新逻辑。
   - 这保证了向后兼容，不会把原先的 validator 工作流打坏。

4. 新增 `policy_route_review` tool
   - 这样 route review 不只是 CLI 可用；
   - 将来 agent 也可以直接把 route plan 作为 tool 调用交给 validator 风格的逻辑去审。

这个文件解决的问题是：

- 之前 route 只有“写入能力”，没有“复核能力”；
- 现在 route 产出后，可以做一致口径的轻量校验，而且不会变成硬门槛。

### `shared/workflows/route_review.prose.md`

这是本轮新增的最小 workflow 示例。

它的作用是：

1. 给 route review 一个可复用样板
   - 未来如果 `main`、`learning`、`curator` 先做了 route 计划，就知道怎么把它交给 validator。

2. 告诉 agent 这项能力应该如何串起来
   - 不是只知道“有工具”，而是知道：
     - route plan 先产出
     - validator 再轻量复核
     - advisory 再反馈给 main

3. 保持软约束
   - 这是样板，不是强制执行器；
   - 所以后续可以基于它演化，而不是被它锁死。

### `workspace-main/MEMORY.md`

这里新增的是一条系统事实：

- 已经存在轻量 `route review loop`
- 并且它是 advisory 级复核，而不是硬审批

作用：

- 让主上下文在组装时就知道这项能力已经可用；
- 避免以后 agent 只能“靠偶然想到”才使用它。

### `workspace-validator/AGENTS.md`

这里补的是 validator 角色边界说明：

- validator 可对 route plan 做轻量复核；
- 但默认以 advisory 为主；
- 不把可探索空间直接判成 FAIL。

作用：

- 让 validator agent 的角色说明与本轮新增能力一致；
- 避免 validator 后续把 route review 错当成强审批。

### `shared/policies/Validation_Rules.md`

这里新增的是 route review 的补充说明。

作用：

1. 明确 route review 默认是轻量复核
2. 明确“可行但未必最优”的 route 方案优先给 advisory
3. 明确 `logs` 默认是建议项，不是硬失败项

它解决的问题是：

- 如果不把这层写进 validation policy，未来 validator 很容易把 route review 越做越硬；
- 现在这份文件把边界钉住了：有复核，但不僵化。

## 四、这轮实现对自迭代意味着什么

本轮新增内容并没有让系统“自动自我改写”。

它做的是更合理的一步：

- 先让 route 分发有一个轻量反馈回路；
- 再让这些 feedback 逐步沉淀进 SOP；
- 以后若某类 advisory 高频重复，再更新 `Conversation_To_Routes_SOP_v1.md`。

这意味着：

- 你要的“先让 agent 自己摸索”是保住了；
- 但摸索不再是完全无反馈、无沉淀的。
