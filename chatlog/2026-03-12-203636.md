# 2026-03-12 对话记录

## 一、本次对话总结

本轮的核心决策有两个：

1. 不把“对话 -> 路由分发”写成硬约束；
2. 只新增一个 SOP，作为 agent 后续自迭代摸索这项能力的参考协议。

围绕这个决策，完成了两件事：

- 判断了当前系统是否已经具备“对话 -> 路由分发”的完整能力；
- 分析了 OpenClaw 当前“自迭代能力”到底是如何实现的，以及我们最近加装的能力会不会破坏它。

结论如下：

- 当前系统已经具备“route 执行能力”，即：
  - 知道有哪些 route；
  - 知道 route 对应什么正式目录和草稿目录；
  - 能按 route 写草稿和生成 patch。
- 但还没有完整的“自然对话自动拆成多个 route 输出并自动调度”的中间层。
- 这意味着当前更准确的状态是：
  - 已有 `route -> 执行`
  - 尚未稳定实现 `对话 -> route -> 执行`

对于“自迭代能力”，本轮的判断是：

- 当前 OpenClaw 的自迭代，本质上不是“自动改写自己核心规则”，而是：
  - 通过记忆累积经验；
  - 通过 workflow + validator 进行输出质量回路；
  - 通过 SOP、流程和人工批准，把有效做法沉淀下来。
- 这是一种“有人类监督的过程迭代”，不是“无监督自我改造”。

因此，我们最近加装的：

- memory-bridge
- obsidian-bridge
- routed draft / patch
- Obsidian 路由表

并不会让系统失去自迭代能力。原因是：

- 这些新增内容大多是“能力扩展”和“可配置工具”；
- 没有去改不可变核心；
- 没有把 route 规则强塞进 immutable 层；
- 没有禁止 agent 在实践中逐步摸索更优的 route 组合。

同时，也要说清楚：

- 我们新增的这些能力本身并**不会自动自迭代**；
- 它们只是“支持自迭代”，不是“自己会迭代”。

也就是说：

- 它们是可学习的基础设施；
- 不是自动反思和自动调参的完整闭环。

## 二、修改了什么文件，作用是什么

### `shared/sop/active/Conversation_To_Routes_SOP_v1.md`

这是本轮唯一新增的业务文件，也是你本轮明确要求“只需要写一个 SOP”对应的落地结果。

它的作用不是强迫系统每次都按固定模板运行，而是：

1. 给 agent 一个默认参考路径
   - 当对话涉及学习系统时，agent 可以参考这个 SOP 去判断：
     - 要不要至少写 `logs`
     - 是否还需要 `knowledge / taskboard / capabilities / reputation / strategy`
     - 是写 draft 还是写 patch

2. 把 route 判断从“纯临场发挥”变成“有协议可复用”
   - 这样后续如果 route 判断效果好，可以继续沿用；
   - 如果发现判断方式不好，也可以直接更新 SOP，而不需要改核心规则。

3. 明确这是一份“软约束”协议
   - 文件头中明确写了：
     - `hard_constraint: false`
   - 这意味着它的定位是“建议路径”，不是“必须照办”。

4. 让后续自迭代有收敛点
   - 如果 agent 在实践中摸索出更好的 route 分发方式，最合理的沉淀位置就是这份 SOP；
   - 这样系统迭代会越来越清晰，而不是散落在临时上下文里。

这个文件具体解决的问题是：

- 之前路由工具已经有了，但缺少一份“什么时候该走哪个 route”的协议说明；
- 现在 route 判断逻辑有了一个统一入口，但仍然保留灵活性。

## 三、当前 OpenClaw 的自迭代能力是如何实现的

### 1. 通过 workflow + validator 做输出回路

关键文件：

- `shared/workflows/learning_review.prose.md`
- `shared/sop/active/Learning_Review_SOP_v1.md`
- `shared/policies/Validation_Rules.md`

这条链路的含义是：

- learning 先根据 SOP 产出日志草稿和状态总结；
- validator 再检查：
  - 是否越权
  - 是否把推测写成事实
  - 是否结构完整
- 如果失败，再修正。

这是一种“输出层自纠偏”，属于最基础的自迭代能力。

### 2. 通过 Task Dispatch + 重试 + 熔断做任务级迭代

关键文件：

- `shared/policies/Core_Routing.md`
- `shared/sop/active/Task_Dispatch_SOP_v1.md`

这条链路的含义是：

- 复杂任务先拆 Task Tree；
- 分配给不同 owner；
- validator 验收；
- 失败则重试；
- 超限则熔断。

这不是让 agent 无限制试错，而是让系统能在“有限重试 + 明确验收”的结构中逐步逼近更好的结果。

### 3. 通过多层记忆做经验累积

关键文件：

- `shared/policies/Memory_Governance.md`
- `extensions/memory-bridge/index.mjs`

这条链路的含义是：

- Working / Semantic / Graph 会累积短期与长期经验；
- Markdown 仍然是真相层；
- 冲突不会自动覆盖，而是进入 staged/conflict 流程。

这意味着系统可以“记住经验”，但不会因为记住经验就擅自改核心规则。

### 4. 通过 SOP 沉淀有效做法

关键文件：

- `shared/sop/active/Learning_Review_SOP_v1.md`
- 本轮新增的 `shared/sop/active/Conversation_To_Routes_SOP_v1.md`

这部分才是你现在最想要的“软迭代”方式：

- 不把每个判断写死在不可变核心里；
- 先让 agent 在实践中跑；
- 跑出稳定模式后，再把经验回写进 SOP。

## 四、最近加装的部分会不会让系统丧失自迭代能力

结论：不会。

原因有 4 个：

1. 我们没有改 `Immutable_Core`
   - `shared/policies/Immutable_Core.md` 仍然禁止 agent 擅自修改核心规则、启用中的 Active SOP、密钥配置等。
   - 所以系统没有从“可控迭代”变成“自我失控修改”。

2. 我们新增的是工具能力，不是绝对强制行为
   - `obsidian-bridge` 新增 route-aware 命令，只是让 agent 有地方可写；
   - 并没有强制每轮对话都必须 route 分发。

3. 我们保留了 draft-first
   - 正式区默认仍然不直接写；
   - 这让系统可以先尝试、先出草稿、先沉淀模式，再决定是否提升到正式层。

4. route 规则目前仍然是可配置、可更新的
   - 这意味着后续如果 route 组合需要调整，不需要推翻整个架构。

## 五、我们加装的部分是否具有自迭代能力

结论：严格说，它们目前是“支持自迭代”，不是“自动自迭代”。

### `memory-bridge`

具有：

- 经验积累能力
- 召回增强能力
- 冲突留痕能力

不具有：

- 自动修改自身治理规则的能力

### `obsidian-bridge`

具有：

- route-aware 写入能力
- patch 生成能力
- 业务板块映射能力

不具有：

- 自动学习“以后应该更偏向哪个 route”的能力

### `Conversation_To_Routes_SOP_v1`

具有：

- 作为后续 route 实践的参考协议
- 作为后续总结经验的沉淀位置

不具有：

- 自动观察历史效果并自己改写自身内容的机制

所以更准确的判断是：

- 当前新增部分是“自迭代兼容的”
- 但还不是“自迭代闭环完整的”

## 六、如何让 agent 知道自己具有这种能力

如果你坚持不加硬约束，那么最合理的方式就是三层软提示：

1. 工具层
   - agent 在工具列表中能看到 routed draft / patch 能力

2. 记忆层
   - `MEMORY.md` 已写明 route baseline 存在

3. SOP 层
   - `Conversation_To_Routes_SOP_v1.md` 告诉它：
     - 何时该 route
     - 何时只写 logs
     - 何时追加其他 route
     - 何时生成 patch

这三层加在一起，已经能让 agent “知道自己有这项能力”，但不会强制死用。
