## 一、本次对话总结

本轮目标是回答“现在如何给 OpenClaw 添加更多模型”，重点不是立刻改配置，而是先把正确的接入顺序、当前状态和推荐路径说清楚。

这次先确认了你当前 live 配置里的模型状态：

- 当前 agent 主模型仍是 `openai-codex/gpt-5.4`
- fallback 也还是 OpenAI 体系内部：
  - `openai/gpt-5.4`
  - `openai/gpt-5.3-codex`
- 当前可见的 configured model 也只有这 3 个
- `Gemini` 目前只用于 web search，不是 agent 主推理模型

同时还确认了当前 CLI 里与模型接入最相关的命令：

- `openclaw models auth ...`
- `openclaw models set ...`
- `openclaw models fallbacks ...`
- `openclaw models aliases ...`
- `openclaw gateway restart`

因此，本轮给出的结论是：

1. 你现在并不是“完全没有多模型入口”
   - OpenClaw 这版已经支持模型 auth、primary model、fallback list、alias 管理

2. 但你当前 live 配置里，多模型能力还没有真正启用
   - 现在只是 OpenAI 内部回退
   - 还没有 Claude / Gemini / Qwen / Kimi / MiniMax / DeepSeek 的 agent 推理接入

3. 最稳的添加顺序，不是一次性接 6 家
   - 先接 2 到 3 个最有价值的 provider
   - 再把它们加入 fallback 链
   - 最后再做模型分工和自动切换策略

本轮还给出了一条更务实的建议：

- 如果你的目标是“减轻 OpenAI 压力 + token 不足时自动切换”
- 与其一开始就直接维护 6 家 provider，不如先做：
  - `Anthropic`：Claude
  - `Moonshot` 或 `MiniMax`：中文/协作补位
  - `Together` 或 `OpenRouter`：承接 DeepSeek / Qwen / Kimi 这类多模型入口

这样做的作用是：

- 先把 fallback 链跑通
- 再逐步细化到每家原生 provider
- 避免一上来就把 auth、计费、配额、模型 ID、路由策略同时搞复杂
